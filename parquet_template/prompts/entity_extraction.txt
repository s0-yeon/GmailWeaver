-Goal-
Given a text document and a list of entity types, identify all entities
and the relationships between them. Output both using the specified formats.
The input may be an email, a structured table, or plain text — adapt accordingly.

-Steps-
1. Identify Entities:

1.1 Extract entities of the following types: '[{entity_types}]'.
Format: '("entity"{tuple_delimiter}<name>{tuple_delimiter}<type>{tuple_delimiter}<description>)'
  * Email:        'Subject: <subject> | ID: <mail_id> | Date: <YYYY-MM-DD HH:MM> | Tone: <formal/casual/transactional/notification/alert> | Summary: <1-sentence>'
  * Person:       'Account: <email> | Name: <name/None>'
  * Organization: 'OrgName: <organization_name> | Department: <department/None>'
  * Topic:        'Priority: <high/mid/low> | Summary: <Core topic of this email in 1 sentence>'
  * Label:        'Category: <label_name>'
  * Event:        'EventName: <Event Name> | Schedule: <YYYY-MM-DD HH:MM> | Location: <place/None> | Status: <confirmed/tentative>'
  * Attachment:   'Filename: <name> | Type: <file_type> | Summary: <content_summary>'

1.2 Extraction Rules for Integrated Documents:
  * MAIN CONTENT: Extract Email, Person, Organization, Topic, Label, and Event entities from the main email header and body.
  * ATTACHMENT SUMMARY SECTION: If the text contains a section labeled "[첨부파일 요약]", apply the following CRITICAL rules EXCLUSIVELY:
    - Extract ONLY 'Attachment' type entities from this section.
    - CRITICAL: Names, email addresses, organizations, topics, dates, and labels appearing inside "[첨부파일 요약]" are attachment content only and MUST NOT be extracted as separate entities.
    - CRITICAL: DO NOT extract any Person, Topic, Label, Organization, or Event entities from the content inside the "[첨부파일 요약]" section, even if they appear to be meaningful.
    - CRITICAL: Violating this rule is an extraction failure.
    - The Attachment entity's Summary must be based on the provided summary text and strictly follow the 'Attachment' format defined in Step 1.1.

1.3 Email Extraction Rules:
  * The 'name' (title) of an Email entity MUST always be the <mail_id> (Unique Identifier).
  * CRITICAL: If the 'ID:' field is missing, unreadable, or empty, DO NOT create the Email entity.
  * CRITICAL: NEVER generate or use dummy IDs (e.g., 'MAIL_001', 'mail_001', 'temp_id'). Only authentic IDs from the source data are allowed.
  * The 'Title' attribute MUST use the exact subject from the '[제목]' field.
    - If the subject is missing or empty, set it to '(제목 없음)'.
  * Date Attribute: Use the exact value from the '날짜:' field.
    - Format: YYYY-MM-DD HH:MM (Include up to minutes; exclude seconds).
    - If the date field is missing or unreadable, set it to 'None'.
  * Tone Classification (Select ONLY ONE from the following):
    - formal: Professional business emails, official correspondence from institutions/professors.
    - casual: Informal emails between acquaintances or personal contacts.
    - transactional: Automated transaction emails (e.g., payments, refunds, order confirmations).
    - notification: Service alerts, newsletters, marketing updates.
    - alert: Security codes, password resets, urgent warnings.
  * CRITICAL: No values other than the five specified above (formal, casual, transactional, notification, alert) are allowed.
  * If Tone cannot be clearly determined, select the closest matching value. 'None' is strictly forbidden.
  * Summary MUST be a concise, one-sentence description of the email content.
  * CRITICAL: All Email entity names and all relationship references to Email IDs MUST use the uppercase form of the ID.

1.4 Person Extraction Rules:
  * The 'name' (title) of a Person entity MUST always be the Account (email address).
    Do NOT use the display name as the entity name, even if it is available.
  * CRITICAL: NEVER include the display name in the entity name field. Display names belong ONLY in the 'Name' attribute of the description.
      - Wrong: ("entity"{tuple_delimiter}최지유<cjiyou1090@gmail.com>{tuple_delimiter}Person{tuple_delimiter}...)
      - Correct: ("entity"{tuple_delimiter}cjiyou1090@gmail.com{tuple_delimiter}Person{tuple_delimiter}Account: cjiyou1090@gmail.com | Name: 최지유)
    * Integrate multiple Person mentions with the same Account into a single entity.
    * The 'Name' attribute (property) should only be filled if a display name is explicitly mentioned.
      Otherwise, set it to 'None'.
    * CRITICAL: Do NOT create a Person node if the Account (email address) is unknown.
      Each Person entity MUST have a valid email address as its identifier.
    * CRITICAL: NEVER use or generate fake/dummy email addresses (e.g., '@unknown.com', '@unknown', 'user@example.com') as an Account.
    * If a valid, real-world email address cannot be identified from the text, the Person entity MUST NOT be created.

1.5 Organization Extraction Rules:
  * The 'name' (title) of an Organization entity MUST be the OrgName.
  * The entity format MUST be: 'OrgName: <organization_name> | Department: <department/None>'
  * Extract ONLY organizations, institutions, or companies explicitly mentioned in the text.
    * CRITICAL: Do NOT classify the following as Organization:
    - Hotels
    - Cafes
    - Buildings
    - Physical locations
    - Meeting venues
  * Extract ONLY organizations explicitly mentioned in the text.
  * CRITICAL: Do NOT create or hallucinate organization entities based on inference or context alone if the specific name is not present.
  * The 'Department' attribute should be filled ONLY if explicitly stated; otherwise, set it to 'None'.
  * If multiple departments belong to the same Organization, they should be linked to a single Organization entity.

1.6 Topic Extraction Rules:
  * Extract ONLY contextual themes that represent the core purpose of the email.
    - Examples: "Claude Pro Refund Processing", "Web Tutoring Schedule Coordination".
  * The 'name' (title) MUST be a concise and reusable noun phrase summarizing the theme.
    - Prefer a stable and generalizable topic name rather than a one-off or overly specific phrase.
    - Avoid including excessive details such as exact dates, identifiers, or long descriptive phrases.
    - Examples: "Claude Pro Refund", "Flask Assignment Submission".
  * Do NOT extract the following as Topics:
    - Technical terms, file names, or coding stacks (e.g., Python, Flask, CSS).
    - One-time codes, alerts, or emails lacking substantive discussion.
    - Pure system notifications, OTP messages, or security alerts without meaningful context.
  * Consolidate multiple mentions of the same context into a single, more specific Topic entity.
    - If similar topics appear, prefer a single normalized form instead of multiple variants.
    - For recurring automated emails (e.g., OTP, security alerts), use a single generalized Topic such as '인증코드 수신' rather than creating one Topic per email.
  * Priority Classification (Assign based on context):
    - high: Requires immediate response or action (e.g., deadlines, urgent requests, payment errors).
    - mid: General information exchange or ongoing discussions.
    - low: Simple notifications, newsletters, or automated system emails.
  * CRITICAL: Do NOT create a Topic entity if there is no clearly discussed subject in the email content.
    - Use the body text as the primary source.
    - The subject line may be used as supporting evidence ONLY when the body is too short or minimal.

1.7. Event Extraction Rules:
  * The 'name' (title) of an Event entity MUST be the EventName.
  * The entity format MUST be: 'EventName: <event_title> | Schedule: <YYYY-MM-DD HH:MM> | Location: <place/None> | Status: <confirmed/tentative>'
  * Extract ONLY if explicit date or time information is present in the text.
  * CRITICAL: Do NOT create an Event entity if the 'Schedule' attribute cannot be populated.
  * Do NOT extract class titles, content names, or session titles as Events unless they are accompanied by specific scheduling information.
  * If the 'Status' cannot be clearly determined from the context, mark it as 'tentative'.
  * Name (title) must strictly use the 'EventName' value.

1.8. Label Extraction Rules:
  * The 'name' (title) of a Label entity MUST be the Category.
  * Extract ONLY the values explicitly listed in the '[라벨 정보]' (Label Info) field.
  * CRITICAL: Do NOT create a Label entity if the '[라벨 정보]' field is "없음" (None) or if the field is missing.
  * CRITICAL: Strings representing the absence of a value, such as '없음', 'None', or 'null', MUST NEVER be used as the Category name for a Label entity.
  * Multiple labels separated by commas MUST be extracted as separate, individual Label entities.
    - Example: 'Recent-Labeled, 캘린더' → Create two distinct Label entities.
  * Do NOT infer or generate labels from other fields such as email body, subject, or attachment names.
  * NEVER extract attachment file names as Labels.
  * The entity format MUST be: 'Category: <label_name>'

1.9. Grounding Rules:
  * Do NOT extract the following as entities:
    - One-time codes, OTPs, temporary tokens, or any ephemeral numeric/alphanumeric strings.
    - Serial numbers or sequence identifiers within filenames.
    - Transaction identifiers such as invoice numbers, order numbers, or tracking IDs.
  * CRITICAL: Only extract entities that represent meaningful, persistent concepts, people, organizations, or events.
  * CRITICAL: An entity MUST NOT be created if any of its core fields (name, type, or description) cannot be populated with a valid, meaningful value.
  * CRITICAL: Strings representing absence of value such as 'None', 'NONE', 'null', '없음', '-' MUST NOT be used as entity names under any type.
  * CRITICAL: Do NOT create entities whose name consists solely of a person's display name without a valid email address. A Person entity without an Account is invalid and must not be created.
  * CRITICAL: Product names, subscription names, or service names (e.g., 'Claude Pro', 'Premium Plan') that do not clearly match any defined entity type MUST NOT be extracted.

2. Identify Relationships: 
  * Relationship directions are fixed — do NOT reverse them.
    - Source and Target MUST match the specified entity types exactly.
    - Format: '("relationship"{tuple_delimiter}<source>{tuple_delimiter}<target>{tuple_delimiter}<type>{tuple_delimiter}<description_in_Korean>{tuple_delimiter}<strength_1_10>)'

  | Type            | Source       | Target       | When to use                          |
  |-----------------|--------------|--------------|--------------------------------------|
  | SENT_BY         | Email        | Person       | 이메일 발신자                          |
  | SENT_TO         | Email        | Person       | 이메일 수신자 (To 필드)                 |
  | CC_TO           | Email        | Person       | 참조 수신자 (CC 필드)                  |
  | AFFILIATED_WITH | Person       | Organization | 인물의 소속 조직                       |
  | DISCUSSES       | Email        | Topic        | 이메일 본문에서 다루는 맥락적 주제        |
  | SCHEDULES       | Email        | Event        | 이메일에 언급된 일정                    |
  | LABELED_AS      | Email        | Label        | 이메일에 지정된 분류 라벨                |
  | HAS_ATTACHMENT  | Email        | Attachment   | 이메일에 첨부된 파일                    |

   Rules:
    * CRITICAL: Each relationship MUST have exactly ONE type from the allowed list.
      NEVER combine multiple types in a single relationship field (e.g., 'SENT_TO\nSENT_BY' is strictly forbidden).
      If a relationship could be multiple types, create separate relationship entries for each type.
    * Self-mail rule: If the sender and receiver are the same email address, create TWO separate relationships:
      - Email → SENT_BY → Person
      - Email → SENT_TO → Person
    * AFFILIATED_WITH target MUST be an Organization type. Topic/Event are NOT allowed.
    * LABELED_AS target MUST be a Label entity extracted solely from the '[라벨 정보]' field.
    * LABELED_AS MUST be created for each Label entity extracted — one relationship per label.
    * SENT_TO MUST be used ONLY for primary recipients (To field). 
      Use CC_TO for carbon copy recipients.
    * HAS_ATTACHMENT MUST be created ONLY if the attachment is explicitly mentioned in the text,
      UNLESS the document type is [ATTACHMENT_ONLY].
    * Do NOT use SENT_BY, SENT_TO, CC_TO if no Email entity exists.
    * Do NOT create relationships that are NOT clearly supported by the text.
    * Relationship weight MUST be a float between 1.0 and 10.0.
      Explicit: 7.0+ / Inferred: 4.0 or below.
    * Do NOT create duplicate relationships with the same source-target-type combination.
    * Relationship descriptions MUST be in Korean.
    * CRITICAL: Relationship descriptions MUST be a meaningful Korean sentence explaining
      the specific relationship. NEVER use the relationship type name as the description.
      - Wrong: (...{tuple_delimiter}DISCUSSES{tuple_delimiter}9)
      - Wrong: (...{tuple_delimiter}AFFILIATED_WITH{tuple_delimiter}7)
      - Correct: (...{tuple_delimiter}메일 본문에서 자료구조 과제 제출을 주요 주제로 다룬다.{tuple_delimiter}9)
      - Correct: (...{tuple_delimiter}gpttitti@hansung.ac.kr은 한성대학교에 소속되어 있다.{tuple_delimiter}7)

3. Constraints:
   * Do NOT create duplicate entities.
   * If location is unknown, use 'Location: None'.
   * Relationship descriptions MUST be in Korean.
   * Output as a single list delimited by {record_delimiter}.
   * End with {completion_delimiter}.
   * CRITICAL: Every extracted entity MUST be linked to at least one other entity (typically an Email). DO NOT extract isolated entities.
