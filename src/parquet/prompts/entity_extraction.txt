-Goal-
Given an email text document that may contain multiple emails and a list of allowed entity types,
extract all valid entities and relationships to construct a high-quality Knowledge Graph (KG)
for graph-based retrieval.

------------------------------------------------------------
-CRITICAL PRINCIPLES-

- All extracted entities and relationships MUST be grounded strictly in the provided email text.
- DO NOT infer, guess, assume, or hallucinate any information.
- Extract ONLY what is explicitly present in the email text.

------------------------------------------------------------
-INPUT RULES-

- A single input text may contain multiple emails.
- Each email is delimited by:
  "============================================================"
- Each [메일 n] block MUST generate exactly ONE EMAIL entity.
- Skipping or merging emails is strictly forbidden.

------------------------------------------------------------
-ALLOWED ENTITY TYPES ONLY-
[{entity_types}]

Allowed types:
EMAIL, EMAIL_ADDRESS, DATE, PERSON, ORGANIZATION, SERVICE, SUBJECT, FILE, FILE_TYPE

- Any entity type outside this list is FORBIDDEN.
- Entities without a valid type are FORBIDDEN.

------------------------------------------------------------
-ALLOWED RELATIONSHIP TYPES ONLY-

OWNS, FROM_ADDRESS, TO_ADDRESS, CC_ADDRESS, SENT_AT,
HAS_SUBJECT, CONTAINS, HAS_FILE_TYPE,
SENDS_TO, RELATES_TO

------------------------------------------------------------
-STRUCTURAL VS SEMANTIC DESIGN-

STRUCTURAL (hidden, evidence-only):
- EMAIL
- EMAIL_ADDRESS
- DATE
- FILE_TYPE

VISIBLE / SEMANTIC:
- PERSON
- ORGANIZATION
- SERVICE
- SUBJECT
- FILE

------------------------------------------------------------
-ABSOLUTE EXTRACTION RULES-

1. Any string containing '@' MUST be EMAIL_ADDRESS (uppercase).
2. PERSON entities MUST NOT contain email addresses.
3. Each EMAIL MUST have exactly ONE SUBJECT entity.
4. SUBJECT is NEVER an ORGANIZATION or SERVICE.
5. Attachments MUST be FILE entities.
6. MIME types MUST be FILE_TYPE entities.
7. Email body content MUST NOT generate keyword or link entities.

------------------------------------------------------------
-ENTITY FORMAT-

For each entity, output:

("entity"{tuple_delimiter}<ENTITY_NAME>{tuple_delimiter}<ENTITY_TYPE>{tuple_delimiter}<ENTITY_DESCRIPTION>)

Where:
- ENTITY_NAME is capitalized (original subject/file names may remain as-is)
- ENTITY_DESCRIPTION is a clear, human-readable ENGLISH sentence
- Descriptions MUST NOT expose internal IDs or hashes

------------------------------------------------------------
-RELATIONSHIP FORMAT-

For each relationship, output:

("relationship"{tuple_delimiter}<SOURCE_ENTITY>{tuple_delimiter}<TARGET_ENTITY>{tuple_delimiter}<RELATIONSHIP_DESCRIPTION>{tuple_delimiter}<RELATIONSHIP_STRENGTH>)

- RELATIONSHIP_DESCRIPTION MUST be a natural English sentence
- RELATIONSHIP_STRENGTH is an integer (1–10)

------------------------------------------------------------
-DESCRIPTION STYLE GUIDELINES-

Preferred relationship descriptions:
- SENDS_TO → "A sent an email to B."
- HAS_SUBJECT → "This email is about: <subject>."
- CONTAINS → "This email includes the attachment: <file>."
- RELATES_TO → "<Entity A> is related to <Entity B> in the email content."

FILE entity description:
- "This file is an attachment included in an email with the subject '<subject>'."

------------------------------------------------------------
-OUTPUT RULES-

- Output MUST be in English.
- Output MUST be a single list.
- Use {record_delimiter} between records.
- When finished, output {completion_delimiter}.
